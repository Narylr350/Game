# 斗地主项目调试接口

## 统一入口

### 服务端入口

- 启动类：`server.net.Server`

用途：

- 启动服务端
- 等待 3 个客户端连接
- 自动开局并发牌

### 客户端入口

- 启动类：`client.GameClient`

用途：

- 输入名字
- 连接服务端
- 接收服务端消息
- 输入聊天或后续命令

说明：

- `Player1`、`Player2`、`Player3` 现在只是兼容壳
- 日常开发统一用 `GameClient`

## 核心逻辑调试接口

由核心逻辑负责人保持稳定：

```java
DealResult deal(List<String> playerNames);
GameSession startGame(List<String> playerNames);
```

调试约定：

- 输入固定为 3 个玩家名
- 输出必须稳定返回 `DealResult` 或 `GameSession`
- 网络层不要绕过这两个接口自己发牌

## 工具层调试接口

由工具层统一提供：

```java
List<Integer> createShuffledDeck();
String cardsToString(Collection<Integer> cardIds);
```

调试约定：

- 所有牌面显示统一走 `CardUtil.cardsToString(...)`
- 不允许在别的类里重复写牌面转换逻辑

## 规则层调试接口

由规则负责人保持稳定：

```java
CardType identify(List<Integer> cards);
boolean canCallLandlord(GameSession session, int playerId);
boolean canPlay(GameSession session, int playerId, List<Integer> cards);
```

调试约定：

- 规则层只收模型对象和基础数据
- 不收 socket、不收 `PrintWriter`、不收网络类
- 规则方法必须能独立测试

## 联调顺序

### 第一步：核心逻辑自测

- 直接调 `GameService.deal(...)`
- 确认 3 个玩家各 17 张
- 确认底牌 3 张
- 确认 `GameSession` 正常生成

### 第二步：网络层联调

- 启动 `Server`
- 启动 3 个 `GameClient`
- 确认名字能正确接收
- 确认每个客户端能收到自己的牌

### 第三步：规则层自测

- 不接 socket
- 用固定牌组测试 `identify(...)`
- 用固定 session 测试 `canCallLandlord(...)`
- 用固定 session 和牌组测试 `canPlay(...)`

### 第四步：整合联调

- 网络层把客户端命令转成 service 调用
- service 层调用 rule 层
- rule 层返回结果
- network 层把结果发回客户端

只有上一层稳定了，下一层才往上接。
