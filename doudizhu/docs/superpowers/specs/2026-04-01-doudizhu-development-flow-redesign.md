# 斗地主项目开发流程重构设计

## 背景

当前项目已经有基础骨架，但对新手来说，理解成本偏高。

主要问题不是“没有分层”，而是“当前阶段的功能还没有长出来，抽象和名词已经先搭好了”。

当前代码和文档中的主要负担有：

- `service / model / rule / net` 的层级概念先于实际收益出现
- `GameSessionManager` 这类中间层在单局流程阶段收益不明显
- 人员分工以“长期负责目录”为主，新手需要先理解边界，再开始做功能
- 文档更强调职责划分，而不是“当前阶段到底先做什么、做成什么算完成”

这会带来两个直接问题：

1. 新手很难顺着主流程理解项目
2. 多人协作容易提前建设未来框架，而不是围绕当前可运行目标推进

## 重构目标

本次重构以两个目标优先：

1. 让新手最快上手
2. 让多人协作时更少冲突

最终产品目标仍然不变：

- 完成一个完整的控制台斗地主

但开发方式改为：

- 功能逐步增加
- 抽象跟着功能出现
- 先跑通当前阶段，再进入下一阶段

## 设计原则

### 1. 先主流程，后抽象

如果一个类或一层结构还没有明确减少复杂度，就先不要拆出来。

判断标准只有一个：

> 如果不用这个类，当前代码会不会明显更乱？

如果答案不明确，就先不加。

### 2. 先单局，后扩展

在控制台斗地主的早期阶段，先只处理：

- 3 个客户端连接
- 单局游戏
- 发牌
- 抢地主
- 出牌
- 结算

多房间、完整生命周期管理、复杂消息协议，不在前期提前建设。

### 3. 结构服务于理解

项目结构应该让人能顺着主流程理解：

连接玩家 -> 创建一局游戏 -> 发牌 -> 抢地主 -> 出牌 -> 结算

而不是先理解大量抽象层，再回头拼主流程。

### 4. 分工服务于阶段目标

多人协作的核心不是“每个人永远守一个目录”，而是“当前阶段所有人都围绕同一个可演示目标推进”。

## 新的项目结构

建议将项目结构收敛为以下几块：

- `client`
  - 控制台客户端入口
  - 玩家输入输出

- `server`
  - 服务端入口
  - 连接管理
  - 消息收发和转发

- `game`
  - 当前单局对局的核心对象和流程
  - 建议放入：
    - `GameRoom`
    - `PlayerState`
    - `DealResult`
    - `GameFlow`

- `rule`
  - 斗地主规则判断
  - 建议放入：
    - `LandlordRule`
    - `PlayRuleChecker`
    - `CardTypeRule`

- `util`
  - 纯工具类
  - 建议保留：
    - `CardUtil`

## 包结构调整建议

### 1. 去掉 `GameSessionManager`

当前 `GameSessionManager` 在单局阶段主要只做：

- 保存 `currentSession`
- 转发 `GameService` 调用

这会增加一层理解成本，但当前没有提供足够收益。

因此建议在本阶段移除，改为由服务端直接持有当前房间或当前对局对象。

### 2. 暂时取消 `service / model` 强拆

在项目前中期，不再强调：

- `model` 只放数据
- `service` 只放流程

因为当前代码里对局状态和对局行为本身就高度耦合，强拆只会制造来回跳转。

因此建议先把这两类内容统一放到 `game` 中，围绕“单局游戏”组织代码。

等后续出现以下信号时，再考虑拆分：

- `game` 下单个文件过大
- 一个类承担了多个明显不同的职责
- 项目进入多局、多房间、断线重连等复杂场景

### 3. `rule` 按功能出现

规则类不再提前搭完整骨架。

改为：

- 开始做抢地主时，再完善 `LandlordRule`
- 开始做出牌时，再完善 `PlayRuleChecker`
- 开始支持更多牌型时，再扩展 `CardTypeRule`

这样规则代码会更贴近当前阶段任务。

## 新的开发流程

开发流程改为“按阶段推进”，每个阶段只围绕一个明确可运行目标。

### 第 1 阶段：连接并发牌

目标：

- 3 个客户端连接服务端
- 服务端创建一局游戏
- 每个玩家收到自己的手牌

当前只需要理解：

- `Server`
- `GameClient`
- `GameRoom`
- `CardUtil`

本阶段不提前建设：

- 抢地主完整状态机
- 出牌规则
- 完整消息协议
- 多房间支持

完成标准：

- 本地能稳定启动服务端和 3 个客户端
- 每个玩家都能看到自己的手牌

### 第 2 阶段：接入抢地主

目标：

- 玩家轮流输入“抢 / 不抢”
- 服务端记录当前轮次和叫地主结果
- 最终确认地主并发到底牌

本阶段新增：

- `LandlordRule`

完成标准：

- 3 个玩家可以完成一轮抢地主
- 地主被正确确认
- 地主收到底牌

### 第 3 阶段：接入基础出牌

目标：

- 玩家按回合出牌
- 先支持基础牌型：
  - 单张
  - 对子
  - 三张
  - 炸弹

本阶段新增：

- `PlayRuleChecker`
- `CardTypeRule` 基础实现

完成标准：

- 回合能轮转
- 非法出牌会被拒绝
- 合法出牌能更新局面

### 第 4 阶段：补全结算与重开

目标：

- 玩家出完牌后判定胜负
- 控制台打印胜负结果
- 可以重新开始下一局

完成标准：

- 一局游戏可完整从连接走到结算
- 支持再次开局

### 第 5 阶段：扩展玩法和体验

在前 4 个阶段稳定后，再逐步增加：

- 更完整的牌型
- 更清晰的控制台提示
- 更规范的消息格式
- 更稳定的异常处理

这一阶段不再是“搭框架”，而是在已有完整主流程上逐步增强。

## 新的多人协作方式

人员分工从“按目录长期驻防”改为“按阶段拿任务”。

这并不否定协作边界，而是把边界改成更适合新手理解的形式。

### 每个阶段只拆三类任务

#### 1. 主流程任务

负责把本阶段真正跑通。

例如第 1 阶段的主流程任务是：

- 从 3 人连接到成功发牌

这是阶段内最重要的任务，需要有一个人负责最终收口。

#### 2. 支撑任务

围绕主流程补齐独立小能力，例如：

- 客户端输入输出优化
- 牌面显示优化
- 控制台提示整理
- 基础测试补充

这类任务更适合新手先上手，因为范围小、结果直观、容易验证。

#### 3. 规则任务

只在当前阶段真的需要时出现。

例如：

- 进入抢地主阶段后，再安排抢地主规则任务
- 进入出牌阶段后，再安排出牌合法性判断任务

这样能避免有人提前实现大量还接不上的未来逻辑。

## 对 `人员分工.md` 的改造方向

`人员分工.md` 目前的分工方式是：

- A 长期负责 `model/service`
- B 长期负责 `net/client`
- C 长期负责 `rule`

这个方式对当前目标不够友好，原因是：

- 新手需要先理解目录边界，才能开始做事
- 目录边界会鼓励“先搭本层框架”，而不是先完成阶段目标
- 主流程被拆散在多个抽象层之间，不利于联调

建议改造为“阶段分工文档”，每个阶段重新列出：

- 当前阶段目标
- 主流程负责人
- 支撑任务列表
- 规则任务列表
- 完成标准

文档结构建议如下：

### 当前阶段目标

写明这一阶段唯一的可演示目标。

### 当前阶段任务拆分

列出：

- 主流程任务
- 支撑任务
- 规则任务

### 当前阶段完成标准

写明满足什么条件后才能进入下一阶段。

### 下一阶段暂不开发内容

明确声明本阶段不提前做什么，防止扩散需求。

## 迁移策略

建议按以下顺序完成迁移：

1. 先改文档
   - 更新项目总览
   - 重写人员分工
   - 明确阶段目标和完成标准

2. 再收代码结构
   - 去掉 `GameSessionManager`
   - 将 `service / model` 向 `game` 合并
   - 调整主流程入口引用

3. 最后补测试和阶段说明
   - 让当前阶段目标可以被清晰验证

这样可以避免“代码先大改，但团队还按旧理解开发”的混乱。

## 非目标

本次重构不追求以下内容：

- 一次性设计完整最终架构
- 提前支持多房间或复杂生命周期
- 在尚未接入功能前先写完整规则系统
- 用过多中间层表达“规范分层”

这些内容都应在功能真正需要时再引入。

## 预期结果

完成本次重构后，项目应具备以下特征：

- 新手可以先顺着主流程理解项目
- 大家知道当前阶段只需要完成什么
- 新功能是逐步长出来的，而不是被未来框架压住
- 协作围绕阶段目标展开，而不是围绕抽象目录展开
- 代码结构仍然保留基本边界，但不会过早细化

## 推荐决策

推荐正式采用以下策略：

- 结构上使用 `client / server / game / rule / util`
- 开发上使用“阶段推进”
- 协作上使用“按阶段拿任务”
- 抽象上坚持“功能到了再拆”

这最符合当前项目“练手 + 多人合作 + 面向新手”的目标。
